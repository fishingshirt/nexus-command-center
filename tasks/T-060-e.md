# T-060-e — Fast-Track & Merge Gate Rules

## Status
`PENDING`

## Goal
Define when a change skips full QA and when it must go through both gates. Also define the merge gate.

## Acceptance Criteria
- [ ] **Fast-Track QC** rule: if commit changes ≤ 20 lines AND touches only 1 file AND the file has ≤2 known dependencies, skip QA and auto-merge if QC passes
- [ ] **Full Pipeline** rule: if commit changes > 50 lines OR touches > 2 files OR touches server JS → must pass QC + QA
- [ ] **Merge Gate**: after QA passes, agent runs `git-merge --ff-only` equivalent (fast-forward merge to main, no squash, no rebase)
- [ ] **Block on FAIL**: nothing can be merged if QC overall = `FAIL` or QA overall = `FAIL`
- [ ] **Fail notification**: if QC or QA fails, send Telegram alert to the user with the failure summary
- [ ] Document these rules in `AGENT_GUIDELINES.md` under "Quality Gate Rules"
- [ ] Make `nexus-push-helper.sh` aware: if `~/.hermes/nexus-quality-queue.json` has FAILED items for the current commit, block push and warn

## Agent Note
Fast-track is dangerous but necessary for quick wins. If a fast-track change later causes a bug, the next audit will catch it and the agent should tighten the fast-track rules. This is a living policy.
