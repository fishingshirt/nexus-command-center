# T-060-e — Fast-Track & Merge Gate Rules

## Status

## Goal
Define when a change skips full QA and when it must go through both gates. Also define the merge gate **and the bounce-back rule on failure**.

## Philosophy
- **Fail fast, bounce back.** If QC or QA finds a bug, the change is not "stuck in the pipeline." It is **sent back to the task workflow** to be fixed.
- There is no "retry in place." The only path after failure is: **fix → re-QC → re-QA → merge.**

## Acceptance Criteria

### 1. Failure = Bounce Back to Task
- [ ] If QC overall = `FAIL`: the change is **NOT kept in the pipeline.** It is treated as a broken, unfinished task. The agent must:
  1. Read the QC failure report.
  2. Go back to the original task file (`tasks/T-XXX.md`).
  3. Fix the bugs.
  4. Re-enter the pipeline from the beginning (QC again).
- [ ] If QA overall = `FAIL`: same bounce-back. The agent must:
  1. Read the QA failure report.
  2. Go back to the original task file.
  3. Fix the bugs.
  4. Re-enter QC → QA from the start.
- [ ] On failure, write to queue: `status: QC-FAILED` or `status: QA-FAILED`, `action: BOUNCE_TO_TASK`, `retry_allowed: true`.

### 2. Fast-Track Rules
- [ ] **Fast-Track QC**: if commit changes ≤ 20 lines AND touches only 1 file AND the file has ≤2 known dependencies → skip QA and auto-merge if QC passes.
- [ ] **Full Pipeline**: if commit changes > 50 lines OR touches > 2 files OR touches server JS → must pass QC + QA.
- [ ] **Medium track**: everything in between → QC + lightweight QA (agent reads checklist, 1-minute smoke test, then sign-off).

### 3. Merge Gate
- [ ] After QA passes, agent runs `nexus-push-helper.sh` which does the actual push.
- [ ] **Block on FAIL**: `nexus-push-helper.sh` Must check `~/.hermes/nexus-quality-queue.json` before pushing. If current commit has a `QC-FAILED` or `QA-FAILED` entry, block the push with an error and instruction to fix the task.
- [ ] No `git push` bypasses the QC/QA queue check.

### 4. Notification
- [ ] On any QC or QA failure, send a Telegram alert to the user with the failure summary and which task was bounced back.

### 5. Documentation
- [ ] These rules added to `AGENT_GUIDELINES.md` under a new section "Quality Gate Rules".

## Agent Note
Fast-track is dangerous but necessary for quick wins. If a fast-track change later causes a bug, the next audit will catch it and the tighten the fast-track rules. This is a living policy. The key principle: **failure always bounces back to the task file. No half-baked code survives.**
