# T-055: Feedback System Overhaul — Agent-Integrated Pipeline

## Status
`PENDING`

## Goal
Replace the current client-only feedback form with a full agent-integrated pipeline. User submissions (feature requests, bugs, improvements, themes) flow to the backend, get classified by the AI agent, and are either auto-converted into whiteboard tasks or queued for manual review. The dashboard gives the user real-time visibility into what happened to their submission.

## Acceptance Criteria

- [ ] **Backend Feedback Queue**
  - Server endpoint: `POST /api/feedback`
    - Body: `{type, title, description, priority, answers[], userId?, timestamp}`
    - Saves to a canonical queue file (e.g., `data/feedback-queue.jsonl`) append-only, one JSON per line.
    - Returns `{ok: true, id}` (id is a timestamp-hash).
  - Server endpoint: `GET /api/feedback?id=...` — read single entry by ID.
  - Server endpoint: `GET /api/feedback/list?status=all|pending|tasked|review|closed` — list submissions.
  - On server startup, if `data/feedback-queue.jsonl` is missing, create it.
  - Validation: reject empty title or description, title max 200 chars.

- [ ] **Agent Wake-Cycle Integration**
  - When the agent wakes and after its normal task work, it checks the feedback queue for unprocessed entries (status = `pending`).
  - Agent classifies each entry into: `feature`, `bug`, `improvement`, `theme`, `other`.
  - Agent evaluates auto-task feasibility using a heuristic:
    - Feature / Improvement: if description is sufficiently detailed (≥ 40 chars) and priority is `must` or `should`, auto-generate a task.
    - Bug: always auto-generate a bug task (`B-XXX`).
    - Theme: if description references colors/vibe/mood, auto-generate.
    - Other / vague: set status = `review` for manual triage.
  - Auto-generation logic:
    - Assign next available task ID / bug ID.
    - Write `tasks/T-XXX.md` with schema: goal, acceptance criteria (populated from answers), status = `PENDING`.
    - Update `WHITEBOARD.md`: append to active task table, bump task count.
    - Commit + push immediately after task creation.
    - Update feedback entry status: `tasked`, set `linkedTask` = `T-XXX`.
    - Sends Telegram notification to user: `"📝 New task T-XXX created from your feedback: {title}"`.
  - If entry is flagged for review, status = `review`, and agent sends Telegram: `"Feedback #{id} needs manual triage. Type 'review feedback {id}' to decide."`.

- [ ] **Dashboard Feedback Center (UI Overhaul)**
  - Keep the existing form (`#view-feedback`) but extend it.
  - After submit: new "Submission Tracker" panel appears below the form.
    - Live list of user's past submissions (fetched from server, not just localStorage).
    - Each row: title, type icon, priority badge, status chip.
    - Status values with distinct colors:
      - `pending` → Yellow/amber chip "Pending Classification"
      - `review` → Purple chip "Under Manual Review"
      - `tasked` → Green chip "Task Created: T-XXX" (clickable link to task file on GitHub)
      - `closed` → Red chip "Closed / Declined"
    - Tap row → expands detail view: full description, agent classification notes, linked task if any.
  - **Notifications integration**: when feedback status changes (agent processes it), a notification toast appears: `"Your feedback '{title}' was converted to T-XXX."`
  - Badge on Feedback nav item shows count of `pending` + `review` submissions.

- [ ] **Settings / Agent Toggle**
  - New toggle in Settings: "Auto-generate tasks from feedback" (default ON).
    - If OFF, agent still classifies but ALWAYS sets status = `review` and waits for manual user confirmation before creating tasks.
  - New toggle: "Notify me when feedback is processed" (default ON).

- [ ] **Security & Data**
  - Feedback queue is stored server-side in a `.gitignore`'d file so it's never committed raw.
  - Only the server process can write to the queue; client POSTs, then polls for status.
  - No PII required; optional user identifier is just a dashboard session hash.

- Empty state on tracker: "No submissions yet. Shape the Nexus above. 💡"

## Server API Endpoints
| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/feedback` | POST | Append submission to queue. Returns `{ok, id}` |
| `/api/feedback/:id` | GET | Read single entry by ID. |
| `/api/feedback/list` | GET | Query params: `status`, `limit`, `offset`. Returns `{items[], total}` |
| `/api/feedback/:id/status` | PATCH | Agent-only (or admin). Body `{status, linkedTask?, note?}` |

## Agent Note
This is a **meta-system task** — it changes how the agent itself ingests user input. Keep the heuristic conservative: when in doubt, flag `review` rather than auto-spamming tasks. The goal is **zero lost user ideas** while maintaining agent discipline. Ensure the wake-cycle code handles feedback BEFORE picking random new tasks, so user requests take priority.
