## Acceptance Criteria
- [ ] **Health check run**: On every wake cycle, verify the local server (`http://localhost:8080`) responds with HTTP 200.
- [ ] **Page load audit**: Confirm each app (Calendar, Notes, To-Do, Weather, Finance, etc.) renders without console errors or blank screens.
- [ ] **Logic sweep**: Re-run the checks from T-038 (calendar save overlap, notes auto-save race, phone-bridge ADB parsing, auth cookie expiry, finance duplicate detection).
- [ ] **Visual regression**: Check for broken layouts, missing CSS, or unstyled flashes (FOUC) across Jarvis and default themes.
- [ ] **Console scan**: Open `/` and each `#app` route; flag any new JS warnings/errors.
- [ ] **Data integrity**: Verify `localStorage` schemas haven't drifted; confirm JSON parses cleanly for all stored keys.
- [ ] **Agent infrastructure**: Ensure WHITEBOARD.md and task files are intact, no broken links, no orphaned `tasks/*.md` references.
- [ ] **Improvement backlog**: If a cycle finds nothing critical, look for micro-improvements (loading states, empty states, keyboard nav, accessibility labels, animation polish).
- [ ] **Task generation**: Every finding gets a task file (`T-051-a`, `T-051-b`, etc.) and a WHITEBOARD entry. Treat them like normal tasks — read, build, verify, commit.
- [ ] **Regression re-verify**: In the cycle after a fix is pushed, re-test the same area to confirm closure.

## Agent Workflow
1. When no `PENDING` or `IN_PROGRESS` tasks remain (only `T-051` and its sub-tasks), this becomes the primary task.
2. Agent runs the checklist above, pushes results to the QA log, and generates tasks for any failures.
3. Agent picks the highest-severity finding next cycle and fixes it.
4. After fixing, agent updates the parent finding to "verify pending".
5. Next cycle: re-verify → close or re-open.
6. If everything passes, agent looks for a micro-improvement (one per cycle) and creates a task for it.

## Tech Notes
- Audit commands are shell / curl based where possible; no Puppeteer required.
- Keep findings lightweight — each sub-task should be SMALL (≤ 50 lines changed) so fixes are fast and safe.
- Severity levels: `CRITICAL` (blocks usage), `HIGH` (broken feature), `MEDIUM` (UX friction), `LOW` (visual polish).
- Use existing bug tracker format in WHITEBOARD.md (`B-###`).

## Dependencies
- All other feature tasks should be `DONE` before this becomes the active primary task.
- Can run in parallel with any ongoing HIGH priority bug fix if no feature work remains.
