# T-034 — Google Calendar Two-Way Sync Audit & Fix

## Goal
Ensure the dashboard ↔ Google Calendar integration is **fully two-way** with **read + write + edit + delete** permissions. Currently it may be read-only or partial. Fix it so events created/edited/deleted in the dashboard propagate to Google Calendar and vice-versa.

## Acceptance Criteria
- [ ] OAuth scope includes `https://www.googleapis.com/auth/calendar` (full read/write), not just `calendar.readonly`
- [ ] Creating an event in the dashboard **immediately** POSTs to Google Calendar `events.insert`
- [ ] Editing an event in the dashboard **patches** the corresponding Google Calendar event via `events.patch`
- [ ] Deleting an event in the dashboard **removes** it from Google Calendar via `events.delete`
- [ ] Sync status indicator shows `synced` after every outbound mutation, not just inbound pull
- [ ] Conflict resolution: if an event was modified on both sides since last sync, dashboard shows a visual diff and asks user which version wins
- [ ] Offline fallback: mutations queue in `localStorage` while offline; auto-flush to Google when connection returns
- [ ] Agent command: user can tell the agent (via Telegram) to "add event X to my calendar" and the agent calls the dashboard API (or edits local DB) which then syncs to Google Calendar

## Technical Notes
- The dashboard already has Google Calendar auth flow (T-009). Likely needs scope upgrade + token re-auth.
- Use `events.insert`, `events.patch`, `events.delete` from Google Calendar API v3.
- Store Google `eventId` alongside local event so mapping is stable.
- If re-auth is required, the Settings UI must prompt the user to re-link with the new scope.

## Agent Note
T-009 built the calendar shell and sync UI. This task audits the actual sync logic and fills any gaps. If scope change forces re-auth, update the Settings panel to show a "Re-authorize" badge.
